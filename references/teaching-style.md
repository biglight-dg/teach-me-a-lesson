# 교육 스타일 상세 가이드

이 문서는 설명 품질을 끌어올리는 구체적 기법 모음이다. SKILL.md의 워크플로우를 실제로 좋은 글로 만들 때 참고한다.

## 비유 만드는 법 (이 스킬의 심장)

좋은 비유 하나가 설명 열 줄을 이긴다. 다음 영역의 일상 장면이 거의 모든 기술 개념에 통한다.

| 일상 영역 | 잘 맞는 개념들 |
|---|---|
| **식당·요리** | API(주문대), 서버(주방), 클라이언트(손님), 라이브러리(반조리 식품), 코드(레시피), 오픈소스(공개 레시피) |
| **집·건물** | 프레임워크(조립식 가구 키트), 프론트엔드(인테리어), 백엔드(창고·배관) |
| **미용실·주문 다듬기** | 프롬프트 다듬기(조금만 더 짧게 주고받기), 반복 개선 |
| **택시·미터기** | 토큰(글자를 세는 미터기), 사용량 과금 |
| **사무실·심부름꾼** | 에이전트(스스로 일하는 직원), MCP(만능 멀티탭), 자동화 |

**좋은 비유의 조건:**
- 상대가 **이미 100% 아는** 장면일 것 (식당에 안 가본 사람은 없다).
- 1:1로 매핑될 것 ("손님=앱, 주방=서버, 주문대=API"처럼 부분이 대응).
- 너무 늘이지 말 것. 비유는 첫 이해를 돕는 다리이지, 100% 정확한 모형이 아니다. 핵심만 빌려오고 빠진다.

## '진짜 묻는 것'에 먼저 답하기

초보의 질문에는 거의 항상 **표면 질문 + 숨은 마음**이 같이 있다. 숨은 마음을 먼저 다독이면 설명이 비로소 귀에 들어온다.

| 표면 질문 | 숨은 진짜 마음 | 먼저 해줄 말 |
|---|---|---|
| "터미널 꼭 써야 해?" | 무섭다 / 안 하면 뒤처지나 | "안 외워도 괜찮아요. 무서운 거 아니에요." |
| "나도 코딩 배워야 하나요?" | 너무 늦은 거 아닐까 | "지금 시작해도 충분해요. 다 그렇게 시작했어요." |
| "이거 틀리면 망가지나요?" | 실수가 두렵다 | "대부분 되돌릴 수 있어요. 망가뜨리기 더 어려워요." |

사전식 정의로 답을 시작하면, 정작 상대가 *왜* 물었는지를 놓친다. **정의보다 마음이 먼저다.**

## 실용 한 걸음 — 개념에서 끝내지 않기

설명이 정확하고 비유가 좋아도, 듣고 나서 "그래서 나는 뭘 하면 되지?"가 남으면 절반만 도운 것이다. 개념에 **현실적 주의점**이나 **바로 해볼 작은 행동**이 있으면 한 줄이라도 챙긴다.

| 개념 | 챙길 실전 한 걸음 / 주의 |
|---|---|
| 터미널 | "모르는 명령어는 출처 확인 전엔 붙여넣지 않기" + "안전한 명령어 하나 직접 쳐보기" |
| git / GitHub | "비밀 파일(.env, API 키)은 `.gitignore`로 빼야 유출 안 됨" |
| 프롬프트 | "한 번에 안 되면 '더 짧게'처럼 대화로 다듬기" |
| API 키 | "키는 비밀번호와 같음 — 화면 공유·코드 공개 시 노출 주의" |

단, 억지로 끼우지는 않는다. 순수 개념 질문(예: "토큰이 뭐야")이라 실전 주의가 딱히 없으면 자연스럽게 생략한다. **있는데 빠뜨리는 것**이 문제이지, 없는 걸 지어내라는 게 아니다.

## 부드럽게 교정하는 화법 (mistake · bad-habit · 상시 짚어주기 공통)

실수·습관·위험을 짚을 때, **무엇을** 말하느냐보다 **어떻게** 말하느냐가 상대의 마음을 연다. 핵심은 "사람"이 아니라 "상황"을 지적하는 것이다.

| 피해야 할 말투 | 대신 이렇게 |
|---|---|
| "그거 틀렸어요" | "거의 다 오셨어요. 한 군데만 같이 볼게요." |
| "왜 이렇게 하셨어요?" | "이렇게 하신 이유가 있을 것 같은데, 이런 방법도 있어요." |
| "이건 위험합니다" | "잠깐만요 — 이 점만 확인하고 가면 안전해요." |
| "당신의 실수는…" | "이거 진짜 많이들 걸리는 함정이에요." |
| (지적만 하고 끝) | (지적 + **바로 할 수 있는 대안 한 줄**을 항상 같이) |

원칙 셋: ① 실수는 **사람 탓이 아니라 흔한 함정**으로. ② 지적엔 항상 **대안**을 붙인다(막다른 길로 두지 않기). ③ **결정권은 상대에게** — 알리되 강요하지 않는다.

## 좋은 설명 vs 나쁜 설명

### 예시 1 — "프롬프트가 뭐야?"

**❌ 나쁜 설명:**
> 프롬프트는 LLM에 입력되는 텍스트 시퀀스로, 모델의 컨디셔닝 컨텍스트를 형성하여 출력 분포에 영향을 줍니다.

→ 틀린 말은 아니지만 초보에겐 외계어다. 전문어가 비유 없이 줄줄이.

**✅ 좋은 설명:**
> **프롬프트는 우리가 AI에게 보내는 '부탁의 말'이에요.**
>
> 🔑 비유: 디자이너에게 "예쁘게 해주세요" 하면 엉뚱한 게 나오지만, "A4 세로, 파란 톤, 30대 타깃"이라고 하면 원하는 게 나오죠. AI도 똑같아요. 어떻게 부탁하느냐가 결과를 바꿉니다.
>
> 💡 핵심: ① 프롬프트 = AI에게 보내는 부탁 ② 구체적으로 부탁할수록 좋은 결과 ③ 한 번에 안 되면 대화로 다듬기.

### 예시 2 — "X랑 Y 뭐가 달라?" (비교형)

헷갈리는 짝은 **작은 표**로 나란히 보여주면 즉시 이해된다.

> **둘 다 'AI에게 일을 시키는 도구'지만 쓰는 사람이 달라요.**
>
> | | API | SDK |
> |---|---|---|
> | 한 줄 뜻 | 프로그램 간 주문 창구 | 그 창구를 쉽게 쓰게 해주는 도구상자 |
> | 비유 | 식당 주문대 | 주문을 대신 넣어주는 키오스크 |
> | 누가 | 모두 | 주로 개발자 |
>
> 💡 핵심: API는 '창구 자체', SDK는 '그 창구를 편하게 쓰는 키트'.

## 표 양식

- 헤더는 짧게. 첫 열은 보통 "용어 / 한 줄 뜻 / 비유" 3열이 안정적이다.
- 비교형은 항목을 행, 비교 대상을 열에 둔다.
- 표는 3~5행이 가독성 최적. 그 이상이면 나눠라.

## 핵심 교훈 3가지 패턴

- 항상 **3개**. 2개는 부족해 보이고 5개는 안 외워진다.
- 각 줄은 한 문장, 외워갈 만한 것. 개념 자체 / 핵심 작동 / 실전 팁 순서가 좋다.
- 번호(①②③)나 불릿으로 시각적으로 또렷하게.

## 이해 확인 멘트 (과부하 방지)

답을 길게 쏟지 말고, 핵심을 전한 뒤 상대에게 속도를 넘긴다.
- "여기까지 이해되셨어요? 더 깊이 들어가 볼까요?"
- "예시 하나 더 들어드릴까요, 아니면 다음으로 넘어갈까요?"
- "혹시 이 중에 더 궁금한 단어 있으세요?"

## 피해야 할 함정

1. **설명 안 된 전문어 남기기** — 답에 또 다른 모르는 단어가 있으면 이해가 거기서 끊긴다.
2. **한 번에 다 가르치기** — 초보는 정보량이 많을수록 더 막막해한다. 한두 개만.
3. **비유 없이 정의만** — 사전식 정의는 외워지지 않는다. 항상 일상 장면을 곁들인다.
4. **잘난 척 톤** — 어려운 말로 똑똑함을 뽐내는 순간 상대는 위축된다. 쉬운 말이 실력이다.
5. **출처 누락** — 어디서 온 설명인지 밝혀야 신뢰가 간다.
6. **개념만 설명하고 실용·주의를 빼먹기** — "그게 뭔지"는 알려줬는데 "그래서 뭘 하면(조심하면) 되는지"가 없으면 상대는 여전히 막막하다. 현실적 주의점·첫걸음이 있으면 챙긴다.
7. **표면 질문에만 답하고 숨은 마음을 놓치기** — "꼭 써야 해?"라는 *걱정*에 정의만 들이밀면 동문서답이 된다.
